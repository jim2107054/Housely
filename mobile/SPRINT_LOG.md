## Agent 1 â€” Auth Stability

The root cause was a race condition between Clerk's `isSignedIn` becoming `true` and the async `syncWithBackend()` call that fetches the user's role from the backend: the navigation effect in `_layout.jsx` fired immediately after sign-in with `user === null` (sync still in flight), so `user?.role === "AGENT"` evaluated to `false` and every owner was incorrectly routed to `/(tabs)` before being corrected once sync resolved â€” producing a visible flicker and redirect loop. The `setTimeout(..., 0)` bandaid did not help because zero-delay still runs before the network response. The fix introduces a hard gate in both `_layout.jsx` and `index.jsx`: `isLoading` is now read from `useAuthStore()`, and any navigation decision that depends on the user's role is skipped entirely while `isSignedIn && isLoading` is true. In `_layout.jsx` this gate lives both as an early-return inside the navigation `useEffect` and as a full-screen `ActivityIndicator` render guard that replaces the navigator tree while sync is in flight, preventing Expo Router from rendering any route (and therefore firing any navigation) until the role is resolved. In `index.jsx` the 3-second splash timer is blocked from starting while sync is pending, so it only counts down once a definitive `user` object (or confirmed signed-out state) is available. The `setTimeout(..., 0)` hack was removed entirely. After the fix, an owner who logs in sees the neutral loading indicator for the duration of the backend sync and is then routed directly to `/(owner)` with no intermediate visit to the user dashboard.

## Agent 4 â€” Location & Nearby

Added GPS auto-detection so the app uses the device's real location on every fresh session instead of always defaulting to Khulna. `detectGpsLocation()` was added to `store/locationStore.js`: it calls `Location.requestForegroundPermissionsAsync()`, and if granted it fetches the device's position with `Accuracy.Balanced`, reverse-geocodes it to a human-readable district/city/country string, and persists it via the existing `setLocation()` action (which also writes to AsyncStorage); if permission is denied or any step throws, it silently returns `{ success: false }` leaving the Khulna default intact with no crash or alert. In `_layout.jsx`, `useLocationStore` is imported alongside a `hasDetectedGps` ref, and a new `useEffect` â€” guarded by `isSignedIn && user && !isLoading && !hasDetectedGps.current` â€” fires `detectGpsLocation()` once per session immediately after the user object is fully resolved (same timing as the socket connect), ensuring GPS detection never races against auth. The nearby screen (`app/(tabs)/nearby.jsx`) was confirmed already correct: it calls `getCoordinates()` from the store on every fetch, so it will automatically reflect the GPS-detected or user-selected location with no additional changes needed. The location selection screen (`app/(location)/index.jsx`) was also confirmed correct: it already implements its own GPS flow for manual override and falls back to "Khulna" as the displayed default city.
